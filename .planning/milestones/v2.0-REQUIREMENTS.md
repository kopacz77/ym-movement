# Requirements Archive: v2.0 YM Wardrobe

**Archived:** 2026-05-30

This is the archived requirements specification for v2.0 YM Wardrobe.

**Defined:** 2026-05-28
**Core Value:** Students can discover, browse, and book lessons from multiple coaches AND browse, fit-match, and rent competition dresses. The super admin manages both coaching operations and the wardrobe marketplace end-to-end.
**Design spec:** [docs/plans/2026-05-28-ym-wardrobe-mvp-design.md](../../docs/plans/2026-05-28-ym-wardrobe-mvp-design.md)

## v1 Requirements (All Shipped)

### Schema

- [x] **SCHEMA-01**: `Dress` model exists with structured sizing (chest/waist/hips/torso min-max), `lengthCm`, `alterableSmaller`/`alterableLarger` flags, all three pricing tiers, security deposit, cleaning fee, owner relation, consignment commission %, status, internal notes, and category/condition enums
- [x] **SCHEMA-02**: `DressImage` model exists with `sortOrder` and `isPrimary` flag, cascading delete from `Dress`
- [x] **SCHEMA-03**: `RentalRequest` model exists with rental type, date range, optional competition info, free-text message, status enum (PENDING/APPROVED/DECLINED/CONVERTED/CANCELED), owner response, and expiry timestamp
- [x] **SCHEMA-04**: `Rental` model exists with snapshot of fees at approval time, payment method, payment status enum, deposit collected/released timestamps, return tracking, condition on return, and consignment payout fields
- [x] **SCHEMA-05**: `Student` model extended with optional body measurement fields (height, chest, waist, hips, torso, inseam, sleeve, preferred fit notes, measurements-updated timestamp) — all nullable
- [x] **SCHEMA-06**: `Settings` model extended with `defaultConsignmentCommissionPct` (default 15), `wardrobeRentalRequestExpiryDays` (default 7), `wardrobeReturnReminderDays` (default 1) — implemented as JSON blob under `Settings.key="wardrobe"` (existing key/value pattern)
- [x] **SCHEMA-07**: Prisma migration applied via `prisma migrate deploy` without affecting any existing production data (6762 rows byte-identical across critical tables)

### Image Storage

- [x] **STORAGE-01**: Vercel Blob signed PUT URL generation via handleUpload route handler at `/api/wardrobe/upload`
- [x] **STORAGE-02**: Client-side image compression (max 1600px wide, 80% quality) via browser-image-compression before upload
- [x] **STORAGE-03**: 8-image cap and 5MB pre-compression cap enforced at 4 layers (UI counter + UI button + attachImage TRPC + route handler token mint)
- [x] **STORAGE-04**: First uploaded image auto-marked primary; admin and consigner can reorder via arrows + re-designate primary via DressImageGallery
- [x] **STORAGE-05**: Image deletion removes Vercel Blob object (`del()`) then DB row in single mutation (delete-blob-first ordering to avoid silent orphan storage cost)

### Catalog & Browse

- [x] **CAT-01**: `/wardrobe` route renders responsive grid (1/2/3/4 cols at sm/md/lg/xl) of dresses with status AVAILABLE or PENDING
- [x] **CAT-02**: Sticky WardrobeFilterBar supports category (multi-select), color (multi-select), size label (multi-select), theme text search, length range slider, and price range slider
- [x] **CAT-03**: Availability date-range filter excludes dresses with overlapping confirmed rentals or approved requests (anti-join against Rental + RentalRequest)
- [x] **CAT-04**: "Fits me" toggle filters by measurement compatibility using stored Student measurements; disabled with tooltip when measurements unset
- [x] **CAT-05**: Sort options: Newest, Price ascending, Price descending, Best Fit (Best Fit only valid when measurements set; BAD_REQUEST otherwise)
- [x] **CAT-06**: Filter state encoded in URL query params so links are shareable; default-value elision keeps `/wardrobe` clean on first load
- [x] **CAT-07**: Best Fit score calculation accounts for `alterableSmaller`/`alterableLarger` slack (`ALTERABLE_SLACK_CM = 2`) and penalizes null structured fields (-0.1 per null, capped at -0.3)
- [x] **CAT-08**: Internal admin notes never exposed — enforced via `PUBLIC_DRESS_SELECT satisfies Prisma.DressSelect` at SQL layer in catalogQueries.ts

### Student Measurements

- [x] **MEASURE-01**: `/wardrobe/measurements` route lets caller view/edit own measurements; supports cm with optional inches input via dual-unit MeasurementForm
- [x] **MEASURE-02**: Measurements persist on `Student` model and surface on dress detail fit comparison via FitCheckCard
- [x] **MEASURE-03**: `measurementsUpdatedAt` timestamp stamped unconditionally on each save (tracks "last reviewed", not "last mutated")

### Dress Detail

- [x] **DETAIL-01**: `/wardrobe/[id]` shows DressImageCarousel, description, CategoryBadge, three pricing tiers via PricingTierTable, and StructuredSizeSummary
- [x] **DETAIL-02**: FitCheckCard shows green/amber/red horizontal bars per measurement (chest/waist/hips) when caller has measurements set; CTA fallback otherwise
- [x] **DETAIL-03**: "Request to Rent" CTA visible to students; replaced with "Sign in" prompt if unauthenticated

### Rental Request Flow (Student)

- [x] **REQUEST-01**: RequestRentalDialog collects rental type radio (Competition/Seasonal/Purchase), date range, optional competition name and date, and required message (≥10 chars)
- [x] **REQUEST-02**: Inline availability check via debounced `wardrobe.requests.checkAvailability` warns student when chosen dates conflict
- [x] **REQUEST-03**: Submit creates `RentalRequest` with status PENDING and fires NOTIFY-04 to dress owner (in-app + email)
- [x] **REQUEST-04**: Student can cancel own PENDING request via MyRentalsView; window.confirm gate + server-authoritative cancel mutation (no optimistic UI per Anti-Pattern D)
- [x] **REQUEST-05**: `/wardrobe/my-rentals` MyRentalsView shows 5 status-bucketed tabs (Pending / Approved-Awaiting-Payment / Active / Past / Declined-Canceled) with URL-synced `?tab=`

### Admin Inventory & Queues

- [x] **ADMIN-01**: `/admin/wardrobe` DressInventoryGrid filterable by all statuses including PENDING_APPROVAL/REJECTED/ARCHIVED
- [x] **ADMIN-02**: `/admin/wardrobe/new` admin creates dress with server-side hardcoded status: AVAILABLE (bypasses approval queue)
- [x] **ADMIN-03**: `/admin/wardrobe/[id]/edit` admin has full field access including internalNotes and consignmentCommissionPct
- [x] **ADMIN-04**: `/admin/wardrobe/requests` RequestQueueTable sorted by competition date with approve/decline + response UI
- [x] **ADMIN-05**: Approve flips PENDING → APPROVED, dress → PENDING; NOTIFY-05 email with payment instructions
- [x] **ADMIN-06**: Decline flips PENDING → DECLINED with owner response; dress returns to AVAILABLE; NOTIFY-05 email
- [x] **ADMIN-07**: `/admin/wardrobe/settings` WardrobeSettingsForm edits defaultConsignmentCommissionPct, expiry days, return reminder days

### Self-Serve Consignment

- [x] **CONSIGN-01**: `/wardrobe/consigned/new` lets any User self-list; status forced to PENDING_APPROVAL server-side
- [x] **CONSIGN-02**: Consigner form hides consignmentCommissionPct, securityDeposit, cleaningFee, internalNotes (FieldVisibility prop on DressFormCore + explicit Prisma select omission in consignerQueries)
- [x] **CONSIGN-03**: Consigner form requires ≥1 image before submit; defense-in-depth at queue-layer filter `Images: {some: {}}` in admin.listPendingApproval
- [x] **CONSIGN-04**: Consigner edit page locks pricing + size after first approval via `lockPricingAndSize` prop driven by `dress.status !== "PENDING_APPROVAL" && dress.status !== "REJECTED"`; runtime gate via `LOCKED_AFTER_APPROVAL_KEYS` const in consignerQueries.update
- [x] **CONSIGN-05**: Consigner can archive own dress only when status is AVAILABLE (assertOwnsDress + status guard in consigner.archive)
- [x] **CONSIGN-06**: `/admin/wardrobe/pending-approval` PendingApprovalQueue with ApproveDressDialog + RejectDressDialog
- [x] **CONSIGN-07**: Admin approve action flips PENDING_APPROVAL → AVAILABLE with optional `consignmentCommissionPctOverride`; clears rejectionReason
- [x] **CONSIGN-08**: Admin reject action flips PENDING_APPROVAL → REJECTED with required reason (1-2000 chars Zod); reason emailed via NOTIFY-03
- [x] **CONSIGN-09**: Rejected dress can be edited (pricing+size unlocked because status is REJECTED) and resubmitted via wardrobe.consigner.resubmit (clears rejectionReason, flips REJECTED → PENDING_APPROVAL, re-fires NOTIFY-01 with isResubmit flag)
- [x] **CONSIGN-10**: `/wardrobe/consigned` ConsignerEarningsTable + MyConsignedDressesList shows owner's dresses across all statuses + per-rental earnings + payout status (5th 'Earnings' tab on MyConsignedDressesList)

### Active Rentals Lifecycle

- [x] **RENTAL-01**: markPaymentReceived converts APPROVED → CONVERTED, creates Rental row, sets dress to RENTED (all in $transaction)
- [x] **RENTAL-02**: Rental snapshots rentalFee/cleaningFee/securityDeposit from Dress at creation time
- [x] **RENTAL-03**: consignmentPayoutAmount calc extracted to pure `src/features/wardrobe/lib/payout.ts`: `rentalFee - round(rentalFee * pct / 100)` when pct > 0, null when pct === 0
- [x] **RENTAL-04**: `/admin/wardrobe/rentals` RentalsTable shows active rentals (3 tabs: Active / Late Fee / Outstanding Payouts) sortable by end date with Returns Due section keyed off wardrobeReturnReminderDays
- [x] **RENTAL-05**: markReturned sets returnedAt + conditionOnReturn, status → RETURNED (does NOT touch Dress.status per Pitfall 8)
- [x] **RENTAL-06**: releaseDeposit sets depositReleasedAt, status → DEPOSIT_RELEASED, dress → AVAILABLE
- [x] **RENTAL-07**: flagLateFee flips paymentStatus → LATE_FEE_OWED for manual resolution; surfaces in Late Fee admin tab
- [x] **RENTAL-08**: markConsignmentPaidOut sets consignmentPaidOut + consignmentPaidOutAt (3 defensive checks: NOT_FOUND, payout-null reject, idempotency reject); fires NOTIFY-09

### Notifications

## Milestone Summary

**Shipped:** 86 of 86 v1 requirements (100% coverage)
**Adjusted during implementation:**
- STORY-01 had two name corrections discovered during Phase 21 research (`MeasurementEditor` → `MeasurementForm`, `ConsignmentEarningsTable` → `ConsignerEarningsTable`) — actual component names took precedence over spec text
- TEST-08 spec path corrected from `tests/e2e/wardrobe.spec.ts` (per design doc) to `tests/wardrobe.spec.ts` (per repo convention)
- SCHEMA-06 implemented as JSON blob under `Settings.key="wardrobe"` rather than new typed columns (matches existing Settings key/value pattern)
- NAV-02 "Consigned" sidebar entry implemented as always-visible rather than gated by "has at least one owned dress" (simpler, no downstream caller pain)
- NOTIFY-07 required ONE schema addition (`Rental.returnReminderSentAt` for idempotency) that wasn't anticipated in the original requirement

**Dropped:** None. Every v1 requirement shipped.

**Net new schema beyond v1 requirements:**
- `Dress.rejectionReason` column added in Phase 18-01 (supports CONSIGN-09 resubmit flow)
- `Rental.returnReminderSentAt` column added in Phase 20-03 (cron idempotency for NOTIFY-07)
