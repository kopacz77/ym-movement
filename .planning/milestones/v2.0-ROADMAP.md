# Milestone v2.0: YM Wardrobe

**Status:** ✅ SHIPPED 2026-05-30
**Phases:** 13-22
**Total Plans:** 47

## Overview

v2.0 adds a competition-dress rental and lightweight consignment marketplace inside YM Movement. Phases 13-22 build it from the schema up: data foundation and image storage first, then admin inventory CRUD, then the marketplace browse experience with body measurements driving fit matching, then the rental request flow, then the active rental lifecycle, then self-serve consignment with admin approval, then payouts, notifications, and a full testing + Storybook sweep at the end. Each phase is independently shippable.

**Design spec:** [docs/plans/2026-05-28-ym-wardrobe-mvp-design.md](../../docs/plans/2026-05-28-ym-wardrobe-mvp-design.md)
**Audit:** [milestones/v2.0-MILESTONE-AUDIT.md](v2.0-MILESTONE-AUDIT.md)

**Key Decisions (excerpt — full log in PROJECT.md):**

- All wardrobe money fields stored as `Int` cents — no Float, prevents accumulated rounding drift
- Database-per-tenant SaaS architecture stays deferred; v2.0 lives inside YM Movement single-org
- Hand-authored Prisma migrations wrapped in `BEGIN;...COMMIT;`, applied via `pnpm prisma:migrate` only (production wipe protection from 2026-04-05 incident)
- `adminProcedure` (not `superAdminProcedure`) on all admin.wardrobe.* — admin-or-super-admin per PERM-02
- Vercel Blob signed-PUT route (handleUpload pattern) for dress images; client-driven attachImage persistence
- Picsum.photos placeholder URLs for seed images (no repo bloat)
- E2E tests verify in-app Notification rows as proxy for "email fired" (no real Resend mock needed in test)
- viteFinal alias for `@/lib/security` + `node:crypto` to unblock Storybook build (tactical fix; architectural server-only split deferred to v2.1)
- Email helpers append to existing `src/lib/email.ts` — no templating library introduced; inline HTML mirroring sendPaymentReminderEmail
- 9 lifecycle email helpers wrapped in non-blocking try/catch at every call site — Resend outage cannot break parent mutation
- vercel.json declares BOTH crons (`send-batch-emails` + new `wardrobe-return-reminders`) at `0 4 * * *` UTC
- Stub-then-swap pattern formalized for parallel-wave plans (Phase 18-04/07, Phase 22-02)
