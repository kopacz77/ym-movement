Export the shared schema:
```
export const createRequestSchema = z.object({
  dressId: z.string().cuid(),
  rentalType: z.nativeEnum(RentalType),
  startDate: z.date(),
  endDate: z.date(),
  competitionName: z.string().max(120).optional(),
  competitionDate: z.date().optional(),
  message: z.string().min(20, "Tell the owner why you're a great match (20+ chars)").max(1000),
}).refine(d => d.endDate > d.startDate, {
  message: "End date must be after start date",
  path: ["endDate"],
}).refine(d => !d.competitionDate || (d.competitionDate >= d.startDate && d.competitionDate <= d.endDate), {
  message: "Competition date must fall inside the rental window",
  path: ["competitionDate"],
});
export type CreateRequestInput = z.infer<typeof createRequestSchema>;
```

Then define `requestsRouter = createTRPCRouter({ ... })` with:

**checkAvailability** (protectedProcedure.query):
- Input: `z.object({ dressId: z.string().cuid(), startDate: z.date(), endDate: z.date() })`
- If `endDate <= startDate`, return `{ available: false, reason: "Invalid date range" as const }` BEFORE hitting DB
- Run `Promise.all` of two `findFirst` queries — Rental with `paymentStatus: { in: ["AWAITING_PAYMENT", "PAID"] }` + overlap predicate, and RentalRequest with `status: "APPROVED"` + overlap predicate. Select `{ id, startDate, endDate }`
- Overlap predicate is EXACTLY: `startDate: { lte: input.endDate }, endDate: { gte: input.startDate }` (canonical interval intersection, mirroring catalogQueries.ts L135-139)
- If either returns a row, return `{ available: false, reason: "Already booked" as const, conflictStart, conflictEnd }`
- Otherwise return `{ available: true, reason: null as const }`

**create** (protectedProcedure.mutation):
- Input: `createRequestSchema`
- Step 1: Resolve caller's Student row: `prisma.student.findUnique({ where: { userId: ctx.session.user.id }, select: { id: true, User: { select: { name: true } } } })`. If null, throw FORBIDDEN "Only students can request a rental".
- Step 2: Resolve the dress: `prisma.dress.findUnique({ where: { id: input.dressId }, select: { id: true, ownerId: true, title: true, status: true } })`. If null, throw NOT_FOUND. If status not in `["AVAILABLE", "PENDING"]`, throw BAD_REQUEST "This dress is not available for new requests". If `dress.ownerId === ctx.session.user.id`, throw BAD_REQUEST "You cannot request your own dress" (Pitfall 9).
- Step 3: Server-side overlap re-check (defense in depth, Pitfall 4). Run the SAME two findFirsts as checkAvailability. If either returns a row, throw BAD_REQUEST "Window no longer available".
- Step 4: `const settings = await getWardrobeSettings(ctx.prisma);` then `const expiresAt = addDays(new Date(), settings.wardrobeRentalRequestExpiryDays);` (Pitfall 10).
- Step 5: `const created = await prisma.rentalRequest.create({ data: { dressId, studentId: student.id, rentalType, startDate, endDate, competitionName, competitionDate, message, status: "PENDING", expiresAt }, select: { id, dressId, status } })`.
- Step 6: Wrap in try/catch — `await createNotification({ userId: dress.ownerId, title: "New rental request", message: `${student.User.name ?? "A student"} requested ${dress.title}`, type: "INFO", link: "/admin/wardrobe/requests" })`. On error: `console.error("[WARDROBE] Failed to notify dress owner:", err)`. Do NOT roll back the request. (Pattern from bookingQueries.ts L400-403.)
- Return `created`.

Inline comment on Step 6: `// notification target is dress owner (Yura or consigner), NOT the requester (Pitfall 5)`.

Use exactly the patterns shown in 16-RESEARCH.md Example 1 — no deviations.

**cancel** (protectedProcedure.mutation):
- Input: `z.object({ requestId: z.string().cuid() })`
- Resolve caller Student: `prisma.student.findUnique({ where: { userId: ctx.session.user.id }, select: { id: true } })`. If null, throw FORBIDDEN.
- Look up the request: `prisma.rentalRequest.findUnique({ where: { id: input.requestId }, select: { id: true, studentId: true, status: true } })`. If null, throw NOT_FOUND.
- PERM-03 enforcement: `if (request.studentId !== student.id) throw new TRPCError({ code: "FORBIDDEN" })`.
- `if (request.status !== "PENDING") throw new TRPCError({ code: "BAD_REQUEST", message: "Only pending requests can be canceled" })`.
- Update and return: `prisma.rentalRequest.update({ where: { id: request.id }, data: { status: "CANCELED" }, select: { id: true, status: true } })`.
- **Critical: do NOT touch Dress.status anywhere in this procedure.** Add an inline comment: `// Pitfall 3: PENDING request cancel does NOT touch Dress.status — only APPROVED requests (Phase 17) hold the dress.` This prevents the regression where a canceled PENDING request flips a correctly-RENTED dress back to AVAILABLE.
