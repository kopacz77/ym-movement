Fix the critical security vulnerability in analytics queries (publicProcedure -> adminProcedure), add coachId scoping to analytics and payment queries, upgrade settings/rink queries to adminProcedure, and wire student bookLesson to inherit coachId from time slots.

Payment scoping must go through the Lesson relation (Payment has no coachId column). Student booking must propagate coachId so lessons are associated with the correct coach.

Output: Five modified query files with proper authorization and coach-scoped data access.
</objective>

  <action>
    **analyticsQueries.ts -- SECURITY FIX:**
    1. Change import: replace `publicProcedure` and `protectedProcedure` with `adminProcedure` from `@/lib/trpc`. Remove unused `publicProcedure` import entirely.
    2. `getOverview`: Change from `publicProcedure` to `adminProcedure`. Add optional input schema: `z.object({ coachId: z.string().optional() }).optional()`. When coachId is provided:
       - `lesson.count` where clause: add `...(input?.coachId && { coachId: input.coachId })`
       - `payment.count` where clause: add `...(input?.coachId && { Lesson: { coachId: input.coachId } })` (Payment has NO direct coachId -- must go through Lesson relation)
       - `payment.aggregate` where clause: same pattern `...(input?.coachId && { Lesson: { coachId: input.coachId } })`
       - `student.count`: Do NOT scope by coachId (students are shared resources)
    3. `getStudentActivity`: Change from `publicProcedure` to `adminProcedure`. Add `coachId: z.string().optional()` to input schema. Add `...(input.coachId && { coachId: input.coachId })` to the lesson findMany where clause.
    4. `getRevenueReport`: Change from `publicProcedure` to `adminProcedure`. Add `coachId: z.string().optional()` to input schema. Add `...(input.coachId && { Lesson: { coachId: input.coachId } })` to the payment findMany where clause. Remember: Payment scoping MUST go through Lesson relation.
    5. `getStudentAttendance`: Change from `protectedProcedure` to `adminProcedure`. Add `coachId: z.string().optional()` to input schema. Add `...(input.coachId && { coachId: input.coachId })` to the lesson findMany where clause.

    **CRITICAL: Payment scoping MUST use `Lesson: { coachId: input.coachId }`, NOT direct `coachId` on Payment. Payment has no coachId column. TypeScript will error if you try `where: { coachId }` on Payment.**

    **paymentQueries.ts changes:**
    Already uses `adminProcedure` (correct). Add coachId filtering via Lesson relation:
    1. `getPayments`: Add `coachId: z.string().optional()` to input schema. Add to the `where` object: `...(input?.coachId && { Lesson: { coachId: input.coachId } })`.
    2. `getPaymentStats`: Accept optional input: `.input(z.object({ coachId: z.string().optional() }).optional())`. Scope all three aggregation queries with `...(input?.coachId && { Lesson: { coachId: input.coachId } })`.
    3. `getPaymentById`, `verifyPayment`, `addPaymentNote`, `sendPaymentReminder`: These operate by payment ID. No coachId scoping needed -- admin operates on specific records.
