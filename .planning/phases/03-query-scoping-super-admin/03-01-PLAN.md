<objective>
Scope all admin schedule queries (time slots, lessons, blocked dates, recurring patterns) by adding optional coachId filtering to reads, setting coachId on creates, and scoping overlap detection per-coach. Upgrade all four files from protectedProcedure to adminProcedure.

Purpose: This is the core data isolation work for SADM-04. Without coachId scoping, a second coach would see Yura's entire schedule and vice versa. Without per-coach overlap detection, Coach B would be blocked from creating slots that overlap with Coach A's slots at the same rink.

Output: Four modified admin schedule query files with coach-aware reads, creates, and overlap checks.
</objective>

    **timeSlotQueries.ts changes:**
    1. Change import from `protectedProcedure` to `adminProcedure`
    2. Replace ALL `protectedProcedure` usages with `adminProcedure` (getTimeSlots, createTimeSlot, deleteTimeSlot, updateTimeSlot, deleteBulkTimeSlots, createBulkTimeSlots)
    3. `getTimeSlots`: Add optional `coachId: z.string().optional()` to input schema. Add `...(input.coachId && { coachId: input.coachId })` to the where clause.
    4. `createTimeSlot`: Add optional `coachId: z.string().optional()` to input schema. Include coachId in the create data: `...(input.coachId && { coachId: input.coachId })`.
    5. `createTimeSlot` overlap detection: Add coachId scoping to the `findFirst` overlap check. If `input.coachId` is provided, add `coachId: input.coachId` to the where clause so only same-coach slots conflict.
    6. `updateTimeSlot` overlap detection: Same pattern -- look up the existing slot's coachId, then add it to the overlap `findFirst` where clause so only same-coach slots conflict.
    7. `createBulkTimeSlots`: Add optional `coachId: z.string().optional()` to input schema. Include coachId in slot data objects. Scope overlap detection per-coach in both the skipOverlapping branch and the regular createMany branch.
    8. `deleteBulkTimeSlots` and `deleteTimeSlot`: No coachId scoping needed on deletes (admin can delete any slot).

    **lessonQueries.ts changes:**
    1. Change import from `protectedProcedure` to `adminProcedure`
    2. Replace ALL `protectedProcedure` usages with `adminProcedure`
    3. `getLessonsByDate`: Add optional `coachId: z.string().optional()` to input schema. Add `...(input.coachId && { coachId: input.coachId })` to the where clause.
    4. `getStudents`: This query fetches students for the assignment dropdown. No coachId scoping needed (students are a shared resource).
    5. `createLesson`: Add optional `coachId: z.string().optional()` to input. Pass it to `prisma.lesson.create` data. Better yet, inherit coachId from the time slot: after fetching the time slot, use `timeSlot.coachId` as the lesson's coachId: `coachId: input.coachId || timeSlot.coachId || undefined`.
    6. `assignStudentToTimeSlot`: After fetching the time slot, inherit coachId into the lesson create: `coachId: timeSlot.coachId || undefined`. This ensures lessons always match their time slot's coach.
    7. `cancelLesson`, `updateLessonType`, `unassignStudent`: These operate by ID. No coachId filter needed -- admin can manage any lesson.

    **IMPORTANT:** Do NOT change `getStudents` to scope by coachId. Students are shared resources accessible to any admin. The research explicitly warns against over-scoping student queries.

    **IMPORTANT:** The `assignStudentToTimeSlot` mutation must inherit coachId from the time slot (`timeSlot.coachId`), NOT from input. The time slot already knows its coach.
