<critical_constraints>

**Database safety (from CLAUDE.md + 2026-04-05 incident):**

- NEVER run `prisma migrate dev`, `prisma migrate reset`, `prisma db push --force-reset`, or `prisma db push` against any database. The only command you may use to apply this migration is `pnpm prisma:migrate` (= `prisma migrate deploy`).
- Run `pnpm db:check` BEFORE invoking `pnpm prisma:migrate` and capture the output. Run `pnpm db:check` AFTER and diff. User, Student, Rink, RinkTimeSlot, Lesson, Payment, Notification counts MUST be identical.
- If `pnpm prisma:migrate` exits non-zero, STOP. Do not attempt to rerun. Do not call `migrate resolve`. Report status to the user and let them decide on Neon PITR vs `migrate resolve --rolled-back`.

**Migration authoring path (manual sequencing — not the usual workflow):**

`prisma migrate deploy` only applies committed migration files; it does NOT generate them. The generation must come from `prisma migrate dev --create-only` (which is the only safe variant of `migrate dev` because it does not apply or reset). After generation, you MUST manually edit the generated `migration.sql` to wrap it in `BEGIN; ... COMMIT;` before running `pnpm prisma:migrate`. If `migrate dev --create-only` is forbidden by user preference, fall back to writing the `migration.sql` by hand from the schema diff.

**Schema requirements (every one is a verifiable fact, not a vibe):**

1. Named relation `@relation("DressOwner", ...)` on BOTH sides of the User ↔ Dress link. On User: `OwnedDresses Dress[] @relation("DressOwner")`. On Dress: `Owner User @relation("DressOwner", fields: [ownerId], references: [id], onDelete: Restrict)`.
2. `DressImage.Dress` uses `onDelete: Cascade`. `RentalRequest.Dress`, `RentalRequest.Student`, `Rental.Dress`, `Rental.Student` all use `onDelete: Restrict`.
3. `Rental.requestId` is `@unique` (one rental per request).
4. Add Student-side relations for `RentalRequest[]` and `Rental[]` so Prisma can resolve the back-references.
5. All 9 new Student columns are nullable (`Int?`, `String?`, `DateTime?`). Do NOT add `@default(...)` to any of them.
6. All `price`, `fee`, `deposit` fields on Dress and Rental are `Int` (cents). Do not use Float. Defaults per design spec lines 77-81 and 228-231.
7. Indexes per design spec: `Dress` has `@@index([status])`, `@@index([category])`, `@@index([ownerId])`. `DressImage` has `@@index([dressId])`. `RentalRequest` has `@@index([dressId])`, `@@index([studentId])`, `@@index([status])`. `Rental` has `@@index([dressId])`, `@@index([studentId])`, `@@index([paymentStatus])`.

</critical_constraints>
