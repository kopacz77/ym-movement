<objective>
Implement wardrobe global settings using the existing `Settings` key/value JSON pattern — NOT new typed columns — because the actual `Settings` model in this codebase is a key/value store, contrary to what the design spec says (see 13-RESEARCH.md Critical Finding #1).

Purpose: SCHEMA-06 requires `defaultConsignmentCommissionPct` (15), `wardrobeRentalRequestExpiryDays` (7), and `wardrobeReturnReminderDays` (1) to be reachable as global defaults so future phases (14 settings UI, 17 rental approval, 20 notifications) can consume them without each one re-deciding defaults. The correct implementation mirrors the existing `operational` / `payment` / `rinkAreas` rows in `settingsQueries.ts` — one new Settings row with `key = "wardrobe"` containing a JSON-serialized object validated by a Zod schema.

Output: A new feature module file at `src/features/admin/api/queries/wardrobeSettingsQueries.ts` exporting `WARDROBE_SETTINGS_KEY`, `wardrobeSettingsSchema`, `getWardrobeSettings`, `updateWardrobeSettings`, and a `wardrobeSettingsRouter` with `get` and `update` admin procedures. The admin router index gains a `wardrobeSettings` sub-router mount.
</objective>

<critical_constraints>

**No schema changes.** This plan does not edit `prisma/schema.prisma`. The wardrobe Settings extension is implemented at the application layer using the existing `Settings` model's key/value shape. If you find yourself wanting to add a column to `Settings`, stop and re-read 13-RESEARCH.md Critical Finding #1.

**Mirror `settingsQueries.ts` exactly.** The existing file demonstrates the canonical pattern: Zod schema → `prisma.settings.findUnique({ where: { key } })` → JSON.parse(value) → fall back to defaults on missing row → `prisma.settings.upsert` with stringified JSON value. Do not invent a new shape.

**Use a single key constant.** Export `WARDROBE_SETTINGS_KEY = "wardrobe" as const` and import it everywhere. This avoids the typo failure mode described in 13-RESEARCH.md Pitfall #3 (one writer uses `"wardrobe"`, another uses `"Wardrobe"`, neither finds the other's row).

**Fail-soft on parse errors.** If a `Settings` row exists with `key = "wardrobe"` but its `value` is invalid JSON or fails Zod validation, `getWardrobeSettings()` must return the defaults rather than throwing. This is consistent with how the existing `getSettings` query handles missing rows. Bad data should not brick the wardrobe.

**Admin-only.** Both `get` and `update` procedures use `adminProcedure`. Consumers in later phases that need the settings on behalf of non-admin callers (e.g. a consigner form pre-filling commission %) should call the helper functions directly from server code, not through the TRPC router.

</critical_constraints>

```ts
// src/features/admin/api/queries/wardrobeSettingsQueries.ts

import { randomUUID } from "node:crypto";
import type { PrismaClient } from "@prisma/client";
import { TRPCError } from "@trpc/server";
import { z } from "zod";
import { adminProcedure, createTRPCRouter } from "@/lib/trpc";

export const WARDROBE_SETTINGS_KEY = "wardrobe" as const;

export const wardrobeSettingsSchema = z.object({
  defaultConsignmentCommissionPct: z.number().int().min(0).max(100).default(15),
  wardrobeRentalRequestExpiryDays: z.number().int().positive().default(7),
  wardrobeReturnReminderDays: z.number().int().positive().default(1),
});

export type WardrobeSettings = z.infer<typeof wardrobeSettingsSchema>;

const DEFAULTS: WardrobeSettings = wardrobeSettingsSchema.parse({});

// Mirror the typing pattern used in settingsQueries.ts — prisma.settings has narrow
// upsert/findUnique signatures that benefit from an explicit cast.
type PrismaWithSettings = PrismaClient & {
  settings: {
    upsert: (args: {
      where: { key: string };
      update: { value: string; updatedAt?: Date };
      create: { id?: string; key: string; value: string; updatedAt?: Date };
    }) => Promise<{ id: string; key: string; value: string; createdAt: Date; updatedAt: Date }>;
    findUnique: (args: {
      where: { key: string };
    }) => Promise<{ id: string; key: string; value: string; createdAt: Date; updatedAt: Date } | null>;
  };
};

export async function getWardrobeSettings(prisma: PrismaClient): Promise<WardrobeSettings> {
  const row = await (prisma as PrismaWithSettings).settings.findUnique({
    where: { key: WARDROBE_SETTINGS_KEY },
  });
  if (!row) return DEFAULTS;
  try {
    return wardrobeSettingsSchema.parse(JSON.parse(row.value));
  } catch {
    // Corrupt JSON or schema-incompatible value — fall back to defaults rather than crash.
    return DEFAULTS;
  }
}

export async function updateWardrobeSettings(
  prisma: PrismaClient,
  patch: Partial<WardrobeSettings>,
): Promise<WardrobeSettings> {
  const current = await getWardrobeSettings(prisma);
  const next = wardrobeSettingsSchema.parse({ ...current, ...patch });
  await (prisma as PrismaWithSettings).settings.upsert({
    where: { key: WARDROBE_SETTINGS_KEY },
    update: { value: JSON.stringify(next), updatedAt: new Date() },
    create: {
      id: randomUUID(),
      key: WARDROBE_SETTINGS_KEY,
      value: JSON.stringify(next),
      updatedAt: new Date(),
    },
  });
  return next;
}

export const wardrobeSettingsRouter = createTRPCRouter({
  get: adminProcedure.query(async ({ ctx }) => {
    try {
      return await getWardrobeSettings(ctx.prisma);
    } catch (error) {
      console.error("Error fetching wardrobe settings:", error);
      throw new TRPCError({
        code: "INTERNAL_SERVER_ERROR",
        message: "Failed to fetch wardrobe settings",
        cause: error,
      });
    }
  }),

  update: adminProcedure
    .input(wardrobeSettingsSchema.partial())
    .mutation(async ({ ctx, input }) => {
      try {
        return await updateWardrobeSettings(ctx.prisma, input);
      } catch (error) {
        console.error("Error updating wardrobe settings:", error);
        throw new TRPCError({
          code: "INTERNAL_SERVER_ERROR",
          message: "Failed to update wardrobe settings",
          cause: error,
        });
      }
    }),
});
```

Notes on the implementation choices:

- `wardrobeSettingsSchema.parse({})` at module load gives us a single source of truth for defaults — if a future engineer changes a `.default(15)` to `.default(20)`, both new-row creation and missing-row fallback see the new value.
- `randomUUID()` for the `id` mirrors `settingsQueries.ts` because the `Settings` model uses `@default(cuid())` but the existing code provides an explicit id on insert. Either works; mirroring is safer.
- The Zod partial input on `update` lets a UI patch a single field without re-sending the whole settings object.
- The narrow `PrismaWithSettings` cast follows the existing settingsQueries.ts pattern — Prisma's generated `Settings` delegate is fully typed but the surrounding file uses explicit casts to keep TRPC inference fast. Match the convention; do not invent.

DO NOT call the helpers from this file at module load. Lazy upsert is correct — the row only appears once an admin or a settings consumer hits `update`. The `get` path always returns defaults if no row exists.
