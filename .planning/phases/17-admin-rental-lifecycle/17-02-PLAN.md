  <name>Task 1: Build RequestResponseDialog</name>
  <files>src/features/wardrobe/components/admin/RequestResponseDialog.tsx</files>
  <action>
    Create a controlled modal component that handles BOTH approve and decline via a `decision: "APPROVE" | "DECLINE"` prop. Single component, two visual modes (approve = emerald button + neutral copy, decline = rose button + reason-required copy).

    ## Imports
    ```typescript
    "use client";
    import { zodResolver } from "@hookform/resolvers/zod";
    import { useForm } from "react-hook-form";
    import { toast } from "sonner";
    import { z } from "zod";
    import { Button } from "@/components/ui/button";
    import {
      Dialog,
      DialogContent,
      DialogFooter,
      DialogHeader,
      DialogTitle,
    } from "@/components/ui/dialog";
    import { Textarea } from "@/components/ui/textarea";
    import { api } from "@/lib/api";
    ```

    ## Schema + types
    ```typescript
    const responseSchema = z.object({
      responseMessage: z.string().min(1, "A message is required").max(1000),
    });
    type ResponseInput = z.infer<typeof responseSchema>;

    export interface RequestResponseDialogProps {
      open: boolean;
      onOpenChange: (open: boolean) => void;
      requestId: string;
      decision: "APPROVE" | "DECLINE";
      dressTitle: string;
      studentName: string;
    }
    ```

    ## Component body
    Mount RHF with `defaultValues: { responseMessage: "" }`. On Dialog close (onOpenChange(false)), call `form.reset({ responseMessage: "" })` to clear stale message between opens (Plan 16-05 ADR pattern).

    Mutation:
    ```typescript
    const utils = api.useUtils();
    const respond = api.admin.wardrobeRequests.respondToRequest.useMutation({
      onSuccess: () => {
        utils.admin.wardrobeRequests.listRequests.invalidate();
        toast.success(
          decision === "APPROVE" ? "Request approved" : "Request declined",
          { description: `Notified ${studentName}.` }
        );
        onOpenChange(false);
        form.reset({ responseMessage: "" });
      },
      onError: (e) => toast.error("Action failed", { description: e.message }),
    });

    const onSubmit = (data: ResponseInput) => {
      respond.mutate({
        requestId,
        decision,
        responseMessage: data.responseMessage,
      });
    };
    ```

    ## JSX
    - Title: `${decision === "APPROVE" ? "Approve" : "Decline"} request for ${dressTitle}` (use navy `#1a3a5c` text).
    - Subtitle paragraph (text-sm text-slate-500): APPROVE → "This will mark the dress as held. The student will get a notification and can send payment.". DECLINE → "The dress will return to available immediately. The student will see your reason."
    - Single Textarea bound via `{...form.register("responseMessage")}`, placeholder differs by mode ("Add an optional note for the student about pickup, fit, etc." vs "Tell the student why you're declining.").
    - Inline error: `{form.formState.errors.responseMessage && <p className="text-xs text-rose-600 mt-1">{form.formState.errors.responseMessage.message}</p>}`.
    - DialogFooter: Cancel button (variant="outline", disabled={respond.isPending}) + Submit button.
    - Submit button className: APPROVE → `bg-emerald-600 hover:bg-emerald-700 text-white`; DECLINE → `bg-rose-600 hover:bg-rose-700 text-white`.
    - Submit label: "Approve" or "Decline". Disabled when `!form.formState.isValid || respond.isPending`.

    ## Constraints (DO NOT)

    - Do NOT use two separate dialogs. The mode discriminator is the entire reason for unification.
    - Do NOT call `respond.mutate` outside `form.handleSubmit(onSubmit)` — Zod validation is the gate.
    - Do NOT inline the Zod schema in JSX scope — keep it module-level so the hook gets a stable resolver reference.
    - Do NOT mark `responseMessage` optional. APPROVE could plausibly be optional but uniformity (server requires it per Plan 17-01's `z.string().min(1)`) avoids per-mode logic AND nudges admins to write helpful pickup notes.
    - Do NOT use sonner's `info` toast — use `success` (APPROVE/DECLINE both completed an action; the differentiation lives in the message body).
    - Do NOT pull a notification preview/echo back to the admin — the mutation's onSuccess + invalidate is the source of truth.
  </action>

    Create the tabbed admin queue component. Two tabs: "Pending" (the queue admins triage) and "Awaiting Payment" (APPROVED requests waiting for the admin to click Mark Paid). Each tab fetches a separate `listRequests` query.

    ## Imports
    ```typescript
    "use client";
    import { format } from "date-fns";
    import { useState } from "react";
    import {
      Tabs,
      TabsContent,
      TabsList,
      TabsTrigger,
    } from "@/components/ui/tabs";
    import {
      Table,
      TableBody,
      TableCell,
      TableHead,
      TableHeader,
      TableRow,
    } from "@/components/ui/table";

    import { api } from "@/lib/api";
    import { CategoryBadge } from "@/features/wardrobe/components/CategoryBadge";

    import { RequestResponseDialog } from "./RequestResponseDialog";
    import { RecordPaymentDialog } from "./RecordPaymentDialog"; // from Plan 17-03
    ```

    ## State
    ```typescript
    type DialogState =
      | { kind: "none" }
      | { kind: "respond"; requestId: string; decision: "APPROVE" | "DECLINE"; dressTitle: string; studentName: string }
      | { kind: "payment"; requestId: string; dressTitle: string; studentName: string };

    const [dialog, setDialog] = useState<DialogState>({ kind: "none" });
    ```

    ## Queries
    ```typescript
    const pendingQuery = api.admin.wardrobeRequests.listRequests.useQuery({ status: "PENDING", page: 1, limit: 50 });
    const approvedQuery = api.admin.wardrobeRequests.listRequests.useQuery({ status: "APPROVED", page: 1, limit: 50 });
    ```

    ## JSX structure

    ```tsx
    <div className="space-y-6">
      <Tabs defaultValue="pending" className="w-full">
        <TabsList>
          <TabsTrigger value="pending">
            Pending {pendingQuery.data && `(${pendingQuery.data.total})`}
          </TabsTrigger>
          <TabsTrigger value="awaiting-payment">
            Awaiting Payment {approvedQuery.data && `(${approvedQuery.data.total})`}
          </TabsTrigger>
        </TabsList>

        <TabsContent value="pending" className="mt-6">
          {pendingQuery.isLoading ? <LoadingState /> :
           !pendingQuery.data?.requests.length ? <EmptyState label="No pending requests" /> :
           <PendingTable requests={pendingQuery.data.requests} onApprove={...} onDecline={...} />}
        </TabsContent>

        <TabsContent value="awaiting-payment" className="mt-6">
          {approvedQuery.isLoading ? <LoadingState /> :
           !approvedQuery.data?.requests.length ? <EmptyState label="No requests awaiting payment" /> :
           <AwaitingPaymentTable requests={approvedQuery.data.requests} onMarkPaid={...} />}
        </TabsContent>
      </Tabs>

      {/* Dialogs (mounted once, controlled via dialog state) */}
      {dialog.kind === "respond" && (
        <RequestResponseDialog
          open={true}
          onOpenChange={(o) => !o && setDialog({ kind: "none" })}
          requestId={dialog.requestId}
          decision={dialog.decision}
          dressTitle={dialog.dressTitle}
          studentName={dialog.studentName}
        />
      )}
      {dialog.kind === "payment" && (
        <RecordPaymentDialog
          open={true}
          onOpenChange={(o) => !o && setDialog({ kind: "none" })}
          requestId={dialog.requestId}
          dressTitle={dialog.dressTitle}
          studentName={dialog.studentName}
        />
      )}
    </div>
    ```

    Actions cell: Two buttons:

    - "Decline" (variant="outline", className="border-rose-200 text-rose-700 hover:bg-rose-50") → same with decision "DECLINE".

    ### AwaitingPaymentTable
    Same columns minus the Approve/Decline actions. Actions cell: single "Mark Paid" button (variant="default", className="bg-cyan-600 hover:bg-cyan-700") → `setDialog({ kind: "payment", requestId: r.id, dressTitle: r.Dress.title, studentName: r.Student.User.name })`.

    ### LoadingState

    ### EmptyState

    ## Constraints (DO NOT)

    - Do NOT collapse PendingTable and AwaitingPaymentTable into one parameterized component — they diverge on actions and on which queries they consume. Two co-located tables, one file.
    - Do NOT add column sorting — sort happens server-side via the listRequests orderBy. Client doesn't override.
