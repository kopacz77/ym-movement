<task type="auto">
  <name>Task 2: Create MeasurementForm component</name>
  <files>src/features/wardrobe/components/MeasurementForm.tsx</files>
  <action>
Create `src/features/wardrobe/components/MeasurementForm.tsx` as a Client Component (`"use client"`).

**Header comment:**
"Student body-measurement editor. Hydrates from wardrobe.measurements.get and persists via wardrobe.measurements.update. cm is canonical (Phase 13 schema columns are Int cm); inches are computed per-field as a read-only helper to the right of each input — NOT stored.

Pattern: RHF + zodResolver, defaults via measurementUpdateSchema.parse({}), rehydrate via form.reset(data) in useEffect when query resolves (avoids remount flicker — see 14-04 ADR)."

**Implementation requirements:**

1. Import the schema from Plan 15-01 to keep validation single-sourced:
   ```ts
   import { measurementUpdateSchema } from "@/features/wardrobe/api/queries/measurementQueries";
   ```
   (If the schema is not exported there, ADD `export` to it in Plan 15-01's file as part of this task.)

2. Use the `useForm<z.input<S>, unknown, z.output<S>>` generics pattern from 14-04 ADR — schema fields are nullable+optional so input/output diverge.

3. Numeric fields use `setValueAs: (v) => (v === "" || v == null ? null : Number.parseInt(String(v), 10))` on register() — empty string maps to null (NEVER 0).

4. Fields (in this rendering order, paired into 2-column rows on md+):
   - heightCm
   - chestCm
   - waistCm
   - hipsCm
   - torsoCm
   - inseamCm
   - sleeveLengthCm
   - preferredFitNotes (textarea, full row)

5. For each numeric field, layout:
   ```
   <label>{Field Name}</label>
   <div className="flex items-center gap-2">
     <Input type="number" step={1} {...register} placeholder="cm" />
     <span className="text-xs text-slate-500 w-16 text-right">{cmToInchesDisplay(value)} in</span>
   </div>
   ```
   `cmToInchesDisplay(cm)` is a local pure helper: `(cm: number | null) => cm == null ? "—" : (cm / 2.54).toFixed(1)`.

6. The current value for the inches display is read via `form.watch("chestCm")` etc. — re-renders on each keystroke (cheap; 9 fields max).

7. Above the form, render an editorial header pattern matching 14-06:
   ```tsx
   <div className="space-y-1">
     <h1 className="text-3xl font-bold tracking-tight text-[#1a3a5c]">Your Measurements</h1>
     <p className="text-sm text-slate-500">
       Used to filter and rank dresses by fit. Last updated: {data?.measurementsUpdatedAt ? format(data.measurementsUpdatedAt, "PPP 'at' p") : "Never"}
     </p>
   </div>
   ```

8. Submit handler:
   ```ts
   const utils = api.useUtils();
   const update = api.wardrobe.measurements.update.useMutation({
     onSuccess: () => {
       utils.wardrobe.measurements.get.invalidate();
       toast.success("Measurements saved");
     },
     onError: (e) => toast.error(e.message),
   });
   const onSubmit = form.handleSubmit((values) => update.mutate(values));
   ```

9. Submit button: cyan `bg-[#0891b2] hover:bg-[#0e7490]` per brand sweep, disabled when `update.isPending`.

10. Form lives in a luxury card: `className="bg-white rounded-xl border border-slate-200 shadow-[0_1px_3px_rgba(0,0,0,0.04),0_8px_32px_rgba(0,0,0,0.02)] p-8"`.

11. Loading state: render a placeholder skeleton when `isLoading`. NOT_FOUND error from get → render a small message "Your student profile is missing — contact your coach" (rare edge case for users with no Student row).

Reuse existing primitives: `Input`, `Textarea`, `Button` from `@/components/ui/*`. Do NOT create new primitives.
