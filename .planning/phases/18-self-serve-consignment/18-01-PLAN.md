<objective>
Add a single nullable column `rejectionReason String?` to the Dress model so that admins can record a required reason when rejecting a consigner-submitted dress (CONSIGN-08), and so the consigner can see WHY their dress was rejected on their edit page (CONSIGN-09 resubmit flow). The column is set by `admin.wardrobe.rejectDress` (Plan 18-02), cleared by `admin.wardrobe.approveDress` and `wardrobe.consigner.resubmit` (Plan 18-02), and read by Plan 18-05's `MyConsignedDressesList`.

Purpose: Phase 18's approval/rejection state machine REQUIRES a place to store the rejection reason that survives across resubmit cycles AND is distinct from `internalNotes` (which is admin-only and explicitly HIDDEN from consigners per CONSIGN-02). Reusing `internalNotes` would leak admin commentary to consigners.

Output: One nullable column added to the Dress table via a hand-authored, transaction-wrapped migration applied via `prisma migrate deploy`. Zero risk to existing data — additive + nullable. After this plan ships, Plan 18-02's TRPC procedures can reference `rejectionReason` in their `data: {}` upsert payloads.
</objective>
